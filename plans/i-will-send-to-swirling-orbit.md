# Fix: Vercel Deploy — pnpm-lock.yaml out of sync with package.json

## Context
Vercel CI uses `pnpm install --frozen-lockfile` by default. The build fails because `pnpm-lock.yaml` still contains 18 lines referencing 6 packages that were removed from `package.json` (Tauri packages, Windows-specific rollup/lightningcss binaries). These were never cleaned up with a fresh `pnpm install` after removal. The lockfile and manifest are out of sync.

**Error:**
```
ERR_PNPM_OUTDATED_LOCKFILE  Cannot install with "frozen-lockfile" because pnpm-lock.yaml is not up to date with package.json
* 6 dependencies were removed: @tauri-apps/api, @tauri-apps/plugin-shell, @rollup/rollup-win32-x64-msvc, @tauri-apps/cli, @tauri-apps/cli-win32-x64-msvc, lightningcss-win32-x64-msvc
```

## Root Cause
`pnpm-lock.yaml` was committed when those Tauri/Windows packages were present. They were later removed from `package.json` but `pnpm install` was never re-run to regenerate the lockfile. CI's frozen-lockfile mode rejects any mismatch.

## Fix (two steps, do both)

### Step 1 — Regenerate the lockfile
Run `pnpm install` in the project root. This rewrites `pnpm-lock.yaml` to match the current `package.json`, removing all orphaned entries. Commit the updated lockfile.

```bash
pnpm install
git add pnpm-lock.yaml
git commit -m "fix: regenerate pnpm-lock.yaml to match current package.json"
```

### Step 2 — Add vercel.json as a safety net
Create `/workspaces/default/code/vercel.json` so future lockfile drift doesn't block deployments:

```json
{
  "installCommand": "pnpm install --no-frozen-lockfile",
  "buildCommand": "pnpm build",
  "outputDirectory": "dist",
  "framework": null
}
```

`outputDirectory: "dist"` and `framework: null` also prevent Vercel from misdetecting the project as Next.js.

## Files to create/modify
- `pnpm-lock.yaml` — regenerated by running `pnpm install` (not manually edited)
- `vercel.json` — new file, created with the config above

## Architecture: React Context + translation keys

### New file: `src/lib/i18n.tsx`
- Export `LangContext` with `{ lang, setLang, t, isRTL }`
- `lang: "en" | "ar"`, persisted in `localStorage("lang")`
- `t(key: string): string` — looks up `translations[lang][key]`, falls back to key
- On `lang` change: set `document.documentElement.dir = isRTL ? "rtl" : "ltr"` and `document.documentElement.lang`
- Export `useLang()` convenience hook
- Export `LangProvider` wrapper component with full EN + AR translation maps

- Active lang: `text-foreground font-semibold`, inactive: `text-muted-foreground`

- Calls `setLang()` from `useLang()`
